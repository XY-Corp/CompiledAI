# Literature Review: The Bitter Lesson and Scaling Laws for Code (2020-2026)

## Key Papers and Research

### 1. Foundational Scaling Laws Papers

**"Scaling Laws for Neural Language Models" (Kaplan et al., OpenAI, Jan 2020)**
- arxiv.org/abs/2001.08361
- Established power-law relationships: loss scales predictably with model size, dataset size, and compute
- Key finding: "Larger models are significantly more sample-efficient"
- Initially suggested prioritizing model size over data

**"Training Compute-Optimal Large Language Models" (Hoffmann et al., DeepMind, 2022) - The Chinchilla Paper**
- Overturned OpenAI's scaling laws, showing optimal ratio is roughly N ~ D (parameters ~ tokens)
- Key insight: Most models were undertrained on data relative to their size
- Compute scales quadratically with data: C ~ D²

**"Scaling Laws for Code: Every Programming Language Matters" (Yang et al., Dec 2025)**
- arxiv.org/abs/2512.13472
- Applies Chinchilla-style scaling laws specifically to code generation
- Shows language-specific scaling behaviors across programming languages

### 2. Code-Specific Models & Scaling

**"Code Llama: Open Foundation Models for Code" (Rozière et al., Meta, Aug 2023)**
- arxiv.org/html/2308.12950v3
- 7B, 13B, 34B, 70B parameter variants
- Key result: Code Llama - Python 7B outperforms Llama 2 70B on HumanEval and MBPP
- Demonstrates starting from general language model (Llama 2) outperforms code-only training
- 70B Instruct achieves 67.8% pass@1 on HumanEval

### 3. The Bitter Lesson Debate

**"The Bitter Lesson" (Rich Sutton, March 2019)**
- incompleteideas.net/IncIdeas/BitterLesson.html
- Core thesis: "General methods that leverage computation are ultimately the most effective"
- Argues hand-crafted features lose to scale + simple algorithms

**"The Bitter Lesson is Misunderstood" (Chakrabarti, Aug 2025)**
- obviouslywrong.org
- Key contrarian argument: C ~ D² means data is the bottleneck, not compute
- "We've basically ingested the entire Internet, and there is no second Internet"
- High-quality training data (~10T tokens) is 2-3 orders of magnitude smaller than total data
- Proposes two paths: **Architects** (better inductive biases, ~20-30% steady gains) vs **Alchemists** (synthetic data, high-variance moonshots)

**"Does the Bitter Lesson Have Limits?" (Breunig, Aug 2025)**
- dbreunig.com
- Stockfish counterexample: A small purpose-built model inside conventional chess engine beat neural network approaches
- HRM (Hierarchical Reasoning Model): 27M parameters beats o3-mini-high on ARC-AGI-1
- Argues practical constraints (cost, speed) favor hybrid approaches

### 4. Karpathy's "Software 2.0" Thesis

**"Software 2.0" (Andrej Karpathy, Nov 2017)**
- karpathy.medium.com/software-2-0-a64152b37c35
- "Neural networks represent the beginning of a fundamental shift in how we develop software"
- Software 1.0 = explicit code; Software 2.0 = weights trained on data
- Correctly predicted transition in vision, speech, games, translation
- "When we develop AGI, it will certainly be written in Software 2.0"

**Follow-up: "Vibe Coding" (Karpathy, Feb 2025)**
- Coined term for letting AI write, refine, debug code via natural language

### 5. AI-Generated Code Statistics

**Google (Sundar Pichai, Oct 2024):**
- "More than a quarter of all new code at Google is now generated by AI"
- Reviewed and accepted by engineers

**Microsoft (Satya Nadella, Apr 2025):**
- ~30% of Microsoft code written by AI

**Anthropic (Dario Amodei, Mar 2025):**
- Predicted 90% of code would be AI-written within 6 months
- Prediction being tracked/debated through end of 2025

### 6. Emergence and Its Critique

**"Are Emergent Abilities of LLMs a Mirage?" (Stanford, 2023)**
- bdtechtalks.com
- Found emergence often artifact of non-linear metrics
- When using linear metrics, performance improves smoothly—no sudden jumps
- Challenges notion that scale produces unpredictable breakthroughs

**"Emergent Abilities in Large Language Models: A Survey" (arxiv, Sept 2025)**
- o1 vs GPT-4o on Codeforces: 89% vs 11%
- Shows inference-time scaling (reasoning) produces dramatic improvements

### 7. Inference-Time Scaling (Test-Time Compute)

**"Learning to Reason with LLMs" (OpenAI, 2024)**
- openai.com/index/learning-to-reason-with-llms/
- o1 model demonstrates inference-time scaling pays off
- IOI 2024: 49th percentile with additional training

**"The State of LLM Reasoning" (Raschka, Mar 2025)**
- "s1: Simple Test-Time Scaling" shows "Wait" tokens improve self-correction
- Inference-time scaling now key dimension alongside training scaling

### 8. Limitations & Need for Architectural Innovation

**"Towards Formal Verification of LLM-Generated Code" (Nov 2025)**
- arxiv.org/abs/2507.13290
- "The reliability of LLM code generation...far from strong enough for mission-critical applications"
- Formal verification requires semantics that LLMs don't inherently provide

**MIT Technology Review Survey (Dec 2025):**
- METR study: Developers believed AI made them 20% faster; objective tests showed 19% **slower**
- Code "smells" (maintenance issues) make up >90% of issues in AI-generated code
- GitClear: Sharp decline in code moved/refactored since 2022; rise in copy-pasted code

## Key Arguments: Scale Alone vs. Architectural Innovation

### Arguments FOR Scale Solving Code Generation:
1. **Benchmark trajectory**: HumanEval went from ~33% (Aug 2024) to >70% (Aug 2025)
2. **Inference-time scaling works**: o1 shows reasoning + more compute = dramatic gains
3. **Real adoption**: 65% of developers use AI tools weekly (Stack Overflow 2025)
4. **Enterprise scale**: 25-30% of Google/Microsoft code now AI-generated

### Arguments AGAINST Scale Alone:
1. **Data wall**: High-quality code data is finite (~10T tokens); C ~ D² means compute without data is "lighting cash on fire"
2. **Stockfish pattern**: Small specialized models can beat general large models
3. **Verification gap**: LLMs produce syntactically correct but semantically unreliable code
4. **Maintenance debt**: AI-generated code increases technical debt, "code smells"
5. **Emergence questioned**: May be metric artifact, not fundamental property
6. **Real productivity unclear**: METR study showed 19% slowdown despite perceived speedup

### The "Compiled AI" Thesis (for your context):
The evidence suggests a **hybrid approach** is emerging:
- **Scale is necessary but not sufficient** for production code generation
- **Verification/compilation** adds the reliability layer LLMs lack
- **Architecture matters**: DeepSeek's efficiency shows algorithmic innovation can leapfrog pure scale
- **Domain specialization** (Code Llama, verified code systems) outperforms general scaling

The Bitter Lesson remains valid longitudinally, but **at any fixed point in time**, architectural innovations (like formal verification integration, hierarchical reasoning, or compiler-guided generation) provide competitive advantages over pure scaling.
